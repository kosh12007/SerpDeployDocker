# Техническая документация: Система проверки уникальности

Данный документ описывает внутреннюю архитектуру, логику работы с данными и алгоритмы системы проверки уникальности текста с использованием кеширования.

## 1. Архитектура компонентов

Система состоит из нескольких уровней:

- **Frontend**:
  - `templates/uniqueness.html`: Основной интерфейс проверки.
  - `templates/dashboard/uniqueness_analytics.html`: Панель глобальной статистики (только для SuperAdmin).
  - `static/js/uniqueness.js`: Логика обработки результатов и визуализации.
- **Backend (API/Routes)**:
  - `app/uniqueness/routes.py`: Эндпоинты `/uniqueness/check` и `/uniqueness/estimate`. Реализует фоновое выполнение задач через `threading`.
  - `app/blueprints/dashboard/dashboard_routes.py`: API для получения статистики дашборда.
- **Logic Core**:
  - `app/uniqueness/checker.py`: Класс `UniquenessChecker`. Реализует токенизацию, лемматизацию (`pymorphy2`), взаимодействие с XMLRiver API и алгоритмы сэмплирования.
- **Data Layer**:
  - `app/db/uniqueness_db.py`: Функции управления задачами, лимитами и кешем шинглов.

## 2. Структура Базы Данных

### Таблица `shingles_cache`

Хранит результаты проверок отдельных фраз (шинглов).

| Колонка        | Тип         | Описание                                           |
| -------------- | ----------- | -------------------------------------------------- |
| `shingle_hash` | `CHAR(40)`  | SHA1 хэш нормализованной фразы (lowercase).        |
| `shingle_len`  | `TINYINT`   | Количество слов в шингле (часть PK).               |
| `found_urls`   | `JSON`      | Список найденных URL с совпадениями.               |
| `is_unique`    | `BOOLEAN`   | Флаг уникальности (true если `found_urls` пуст).   |
| `last_updated` | `TIMESTAMP` | Дата последнего обновления (используется для TTL). |

**Первичный ключ**: `(shingle_hash, shingle_len)` — позволяет хранить разные длины шинглов параллельно (Multi-layer Cache).

## 3. Режимы выборки шинглов (Sampling Modes)

Система поддерживает два режима выбора фраз для отправки в платный API, настраиваемых параметром `UNIQUENESS_SAMPLING_MODE`.

### Последовательный (Sequential)

- **Алгоритм**: `all_shingles[::stride]`
- **Плюсы**: Предсказуемое распределение.
- **Минусы**: «Эффект сдвига». Изменение одного слова в начале текста меняет индексы всех последующих фраз, делая кеш бесполезным для фрагментов.

### Детерминированный (DSS - Deterministic Shingle Sampling) — Рекомендуемый

- **Алгоритм**: `int(hash(shingle), 16) % stride == 0`
- **Плюсы**: **Стабильность кеша**. Одна и та же фраза всегда либо отправляется в API, либо нет, независимо от её позиции в тексте.
- **Результат**: Идеальное попадание в кеш при проверке фрагментов одного и того же текста.

## 4. Расчет Аналитики и Эффективности

Метрики рассчитываются в реальном времени при запросе к дашборду через метод `get_uniqueness_analytics` в `uniqueness_db.py`.

### 4.1. Источники данных и алгоритмы расчета

Данные агрегируются из двух основных таблиц:

1.  **`uniqueness_tasks`**: Содержит историю всех завершенных проверок.
    - **Всего проверок (`total_tasks`)**: `COUNT(*)` всех записей со статусом `completed`.
    - **Шинглов проверено (`total_shingles_checked`)**: `SUM(progress_total)` — общее количество фраз (шинглов), на которые был разбит текст перед проверкой.
    - **Сэкономлено лимитов (`total_limits_saved`)**: `total_shingles_checked - SUM(reserved_limits)`. Разница между общим объемом и фактически оплаченными запросами. Значение `reserved_limits` корректируется в конце каждой задачи: из него вычитаются все фразы, найденные в кеше, а также те, которые не удалось проверить из-за ошибок API. Это гарантирует точность статистики экономии.
    - **Cache Hit Rate**: `(total_limits_saved / total_shingles_checked) * 100%`. Процент контента, обслуженного кешем.

2.  **`shingles_cache`**: Содержит базу знаний о фразах.
    - **Актуальные / Просроченные**: Используется **TTL (Time To Live) равный 14 дням**. Сравнивается дата `last_updated` с `NOW() - INTERVAL 14 DAY`. Просроченные записи игнорируются при проверке и обновляются при новом запросе.

### 4.2. Алгоритм поиска ТОП-доменов с дубликатами

Это агрегированная метрика, позволяющая выявить основные сайты-источники заимствований:

1.  **Выборка**: Система запрашивает последние **2000 записей** из кеша, где `is_unique = 0` (найдены совпадения).
2.  **Парсинг**: Для каждого результата извлекается поле `found_urls` (JSON-список ссылок).
3.  **Агрегация**: В коде Python (а не в SQL) происходит парсинг URL с помощью `urllib.parse.netloc` для выделения основного домена.
4.  **Сортировка**: Домены сортируются по частоте появления, и на дашборд выводятся **ТОП-10**.

### 4.3. Оптимизация хранения результатов

Для предотвращения раздувания базы данных и ускорения интерфейса при проверке больших текстов применяется фильтрация результатов:

1.  **Минимальный процент совпадения (`UNIQUENESS_MIN_MATCH_PERCENT`)**: В итоговый отчет попадают только те сайты, на которых найдено более X% контента. Это отсекает случайные совпадения и «мусорные» сайты.
2.  **Максимальное количество сайтов (`UNIQUENESS_MAX_MATCH_URLS`)**: Система сохраняет только ТОП-X сайтов с наибольшим процентом совпадений. Это гарантирует стабильную работу браузера при раскрытии списка доменов.
3.  **Влияние на оценку**: Данная фильтрация влияет только на **детализацию отчета** (список сайтов и подсветка). Итоговый процент уникальности текста по-прежнему рассчитывается на основе всех найденных совпадений.

## 5. Рекомендации по улучшению

Для дальнейшего развития системы аналитики рекомендуется:

1.  **График динамики (Time Series)**: Добавить визуализацию изменения `Hit Rate` по дням для отслеживания скорости "обучения" кеша.
2.  **Экономия в валюте**: Выводить сумму сэкономленных средств на основе стоимости лимитов XMLRiver.
3.  **Теги контента**: Внедрить категоризацию задач (например, "Статьи", "Карточки"), чтобы видеть эффективность кеширования для разных типов текстов.
4.  **Мониторинг объема**: Выводить физический размер таблицы `shingles_cache` в МБ для контроля роста БД.
5.  **Инструменты обслуживания**: Добавить механизмы принудительной перепроверки просроченных записей для популярных доменов.
