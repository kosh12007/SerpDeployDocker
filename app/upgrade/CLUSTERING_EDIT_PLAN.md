# План реализации модуля редактирования кластеров

## 1. Backend изменения (Flask + MySQL)
- **Endpoint `POST /api/clusters/move-keyword`**: 
  - Принимает: `keyword_id`, `target_group_id`.
  - Действие: Обновляет `group_id` в таблице `keywords`.
- **Endpoint `POST /api/clusters/merge`**:
  - Принимает: `source_group_id`, `target_group_id`.
  - Действие: Переносит все ключевые слова из source в target, затем удаляет source_group.
- **Endpoint `POST /api/clusters/rename`**:
  - Принимает: `group_id`, `new_name`.
  - Действие: Обновляет `name` в таблице `groups`.
- **Endpoint `POST /api/clusters/delete`**:
  - Принимает: `group_id`.
  - Действие: Удаляет группу и сбрасывает `group_id` у связанных ключевых слов в NULL.

## 2. Frontend изменения (Tailwind + JS)
- **Библиотека**: Рекомендуется `SortableJS` для легкой реализации Drag-and-Drop.
- **Компоненты**:
    - Контейнер групп (Grid).
    - Карточка группы (Droppable zone).
    - Элемент ключевого слова (Draggable item).
- **Индикация**: Обновление суммарной частотности кластера (Total Volume) на лету через JS при переносе слов.

## 3. UI/UX элементы
- Кнопка "Создать пустую группу" (для ручной сортировки).
- Кнопка "Удалить группу" (корзина на карточке).
- Input для редактирования названия группы прямо в заголовке карточки.
2. Подробное ТЗ для реализации функционала
Задача: Создать интерфейс управления семантическим ядром после автоматической кластеризации.

Основные сценарии:

Перенос запроса: Пользователь видит запрос в "Группе А", который больше подходит "Группе Б". Он перетаскивает его мышкой. Происходит мгновенный AJAX-запрос к API.

Оптимизация названий: Алгоритм называет группу по самому частотному запросу, но пользователь должен иметь возможность переименовать её в более понятную (например, "Купить куртки" -> "Коммерческие куртки Москва").

Работа с "некластеризованным": Все слова с group_id IS NULL должны отображаться в отдельной боковой панели или первой карточке, чтобы их можно было разнести по группам вручную.
